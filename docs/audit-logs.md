# 后台操作日志：通用实现方法论

> 适用于管理后台中「谁在什么时间对什么资源做了什么」的追溯与审计场景。

---

## 一、先分清两类日志

| 类型         | 回答的问题                           | 记录位置      |
| ------------ | ------------------------------------ | ------------- |
| **访问日志** | 谁访问了哪个接口、耗时多少、是否报错 | 网关 / 中间件 |
| **操作审计** | 谁修改了哪条业务数据、改了什么       | 业务层        |

不要用访问日志替代操作审计。前者看流量，后者看业务变更，职责不同，应并存。

---

## 二、核心原则

### 1. 审计属于业务，不属于 HTTP

操作语义（「删除了商品 A」）只有业务层知道，HTTP 层只能看到「发了一个 DELETE 请求」。因此：

- 审计写在**业务服务层**，不依赖全局拦截器猜意图
- 接口层只做鉴权、校验、转发，不承担审计职责

### 2. 约定优于散落

统一三件事：

- **动作标识**：`资源.操作` 格式（如 `order.cancel`），集中注册，不散落在各处
- **摘要文案**：一句人话，列表页直接展示
- **变更快照**：结构化存储变更前后状态，详情页可对比

### 3. 只追加，不可改

审计记录一旦写入，不允许修改或删除。这是审计可信度的底线。

### 4. 扩展靠注册，不靠改表

新增管理模块时，理想状态是：

- 加动作注册
- 在对应业务的写操作成功后记一条日志
- 查询页、对比页自动兼容

不应每加一个模块就改表结构、改查询逻辑、改展示页面。

---

## 三、推荐架构

```
用户操作
  → 接口层（薄）
  → 业务服务层（执行业务 + 触发审计）
  → 审计服务（统一写入）
  → 审计存储（只追加）
```

**业务服务层**负责：读旧数据 → 执行变更 → 写新数据 → 调用审计服务。

**审计服务**负责：接收标准化输入，写入存储，对外提供查询。全系统只有一个写入入口。

---

## 四、数据怎么存

建议字段分三层，兼顾可读与可追溯：

| 层次       | 存什么                                | 给谁看                      |
| ---------- | ------------------------------------- | --------------------------- |
| **索引层** | 操作人、动作、资源类型、资源 ID、时间 | 筛选、检索                  |
| **摘要层** | 一句自然语言描述                      | 列表页快速浏览              |
| **详情层** | 变更前/后的结构化快照                 | 详情页、diff 对比、合规举证 |

不需要把每个业务字段都拆成表列。索引 + 摘要 + JSON 快照，足以覆盖绝大多数场景。

### metadata 约定

| 操作 | 快照内容          |
| ---- | ----------------- |
| 新增 | 只存变更后        |
| 修改 | 存变更前 + 变更后 |
| 删除 | 只存变更前        |

这一约定一旦确立，前端的变更对比可以做成通用能力，不绑定具体业务。

---

## 五、什么时候记、记什么

- **只记写操作**：新增、修改、删除。查询类不记，避免噪音
- **只记成功**：失败或未生效的变更不入账
- **在业务完成之后记**：先完成变更，再写审计（理想情况下与业务同一事务提交）

摘要写法建议统一：**动词 + 对象类型 + 名称或 ID**，保持可搜索、可读。

---

## 六、展示层思路

### 列表页

展示：时间、操作人、操作类型、摘要。筛选维度：模块、操作类型、关键词、时间范围。操作类型的中文标签从注册表读取，不在页面硬编码。

### 详情页

- 有变更前后快照 → 左右对比（类 Git diff：左旧右新，差异高亮）
- 无标准快照 → 回退展示原始数据

diff 组件应设计为**通用的**：只认「变更前 / 变更后」结构，不认具体业务。字段中文名可作为可选增强，不影响核心能力。

---

## 七、新增业务模块的标准流程

1. **注册动作**：在统一注册表中声明该模块的 create / update / delete
2. **服务层埋点**：所有写操作经过业务服务，成功后调用审计服务
3. **接口层保持薄**：不直接写库，不写审计
4. **（可选）补充字段标签**：改善 diff 页面的中文可读性

查询页、审计表、diff 组件均不需要改动。

---

## 八、常见方案对比

| 方案                  | 优点               | 缺点                     | 适用                   |
| --------------------- | ------------------ | ------------------------ | ---------------------- |
| HTTP 拦截器自动记     | 省事，不易漏       | 语义不准，拿不到变更内容 | 原型 / 小 demo         |
| 业务层手动记          | 语义准确，快照完整 | 依赖开发者记得写         | 中小型项目（推荐起点） |
| 封装写操作 + 统一埋点 | 减少重复和遗漏     | 需统一服务层入口         | 模块增多后             |
| 领域事件 + 审计消费者 | 业务与审计完全解耦 | 复杂度高                 | 多团队 / 大型系统      |

演进路径建议：**手动记 → 统一封装 → 事件驱动**，按项目规模逐步升级，不要一开始过度设计。

---

## 九、必须提前想清楚的边界

| 问题         | 要想好的事                                       |
| ------------ | ------------------------------------------------ |
| **权限**     | 谁能看全部日志？普通用户能看自己的吗？           |
| **可靠性**   | 业务和审计是否同一事务？审计写失败怎么办？       |
| **防遗漏**   | 靠什么保证每个写操作都记了？规范、封装还是测试？ |
| **保留策略** | 日志保留多久？是否需要归档？                     |
| **敏感数据** | 快照里是否包含密码、Token 等？需要脱敏吗？       |

这些不影响第一版上线，但决定系统能否从「能用」变成「可审计」。

---

## 十、自检清单

上线前对照：

- [ ] 所有写操作都经过业务服务层，而非散落在接口里
- [ ] 审计只有一个写入入口
- [ ] 动作标识有统一注册表
- [ ] metadata 遵循 before / after 约定
- [ ] 审计记录不可修改、不可删除
- [ ] 列表页筛选项来自注册表，非硬编码
- [ ] 详情页支持通用 diff 对比
- [ ] 明确了日志的查看权限

---

## 一句话总结

> 操作日志的本质是**业务契约**：由业务服务在变更成功后，通过统一审计服务，以「索引 + 摘要 + 快照」三层结构只追加写入。扩展新模块靠动作注册和 metadata 约定，而不是改表、改页面、改拦截器。
